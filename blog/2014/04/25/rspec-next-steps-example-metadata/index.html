RSpec Next Steps: Example Metadata

Apr 25th, 2014

One of my favourite Ruby gems RSpec, the testing library. Coming from JUnit and derived test suites it was a real breath of fresh air. It’s syntax is a real winner:

This is a trivial example, but you would be surprised how many new developers understand this over a more traditional JUnit style. Even though it doesn’t look like a Ruby file most seem to understand RSpec almost intuitively. It’s almost as though RSpec is a magic black box that just runs your examples.

But sometimes I need to take a look under the hood, and that is where example metadata comes into play. Example metadata is a Hash available within your it block. It contains a lot of information about the current example that is running. It is simple enough to find, just put example.metadata inside your test.

This is what a typical metadata Hash looks like:

It might not look like much. You can actually get a lot of these things directly from the example object, like example.full_description which has the complete test description or example.location which contains the file name and line number for the test.

But let’s see what happens with tags. Let’s pretend our test suite has grown dramatically, and one of the specs has become notorious for failing sporadically and requiring tons of developer time to fix up. We’ll add a times_fixed tag to say how many times the spec has been fixed and hours_wasted to say long we have spent on it:

Now where can we find this tag data at runtime? In example.metadata, of course:

Now we can vent our frustration! Let’s add up all the hours wasted in a custom RSpec formatter:

This is where a lot of extensibility in RSpec comes from. Developers can add tags to their specs, those tags are stored in the example.metadata and can be read and manipulated from there. So anytime you want to work with tags, remember example.metadata!
